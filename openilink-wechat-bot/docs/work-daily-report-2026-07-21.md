## 一、语音识别与语音回复链路异常

### 1. 遇到的问题、现象和错误关键词

用户希望机器人在开启“语音回复模式”后直接返回语音，但实际出现了三类现象：

- 微信聊天框没有收到气泡语音，日志却显示“已通过语音回复用户”。
- 语音识别有时完全失败，机器人无法理解语音内容。
- TTS 生成 MP3 时失败，系统退回文字回复。

典型错误关键词：

```text
ffmpeg 未安装或不可用，跳过语音转码
语音识别 API 失败：status=500
SILK 已本地解码为 WAV 后识别
语音识别成功 model=FunAudioLLM/SenseVoiceSmall
sample rate of mp3 should be 32000hz or 44100hz
准备发送微信语音
encodeType=4
已通过语音回复用户
```

其中，`status=500` 说明语音识别服务端没有正常处理请求；`sample rate of mp3...` 是 TTS 服务对 MP3 采样率的明确限制；“日志显示发送成功但聊天框无气泡”说明 SDK 调用返回成功并不等于微信客户端一定按语音消息渲染。

### 2. 分析过程与思考方式

首先把语音链路拆成四个独立阶段：

1. 微信下载 SILK 原始语音。
2. SILK 解码为 WAV/PCM。
3. STT 服务将音频转换为文本。
4. LLM 回复经过 TTS，再通过微信 SDK 发送。

根据日志判断，接收方向已经可以通过本地 SILK 解码得到 WAV，并且 `SenseVoiceSmall` 能成功识别，因此“语音无法识别”不能简单归因于模型能力不足，主要问题是原始 SILK 直接上传时依赖 `ffmpeg`，而运行环境没有安装或无法调用 `ffmpeg`。

发送方向则需要区分“编码成功”和“微信消息类型正确”。生成一个名为 `.silk` 的文件，不代表它一定具备微信语音消息所需的编码参数、采样率、播放时长和协议元数据。日志中的 `encodeType`、`sampleRate`、`playTime` 需要与实际音频内容一致，微信客户端也可能拒绝不符合协议的语音气泡渲染。

### 3. 解决方案

- 接收语音：优先使用项目中的 SILK 编码库本地解码为 WAV，再调用 STT，减少对 `ffmpeg` 的依赖。
- 语音识别：继续使用 `FunAudioLLM/SenseVoiceSmall`，识别成功后只将文本作为内部路由输入，不把“识别结果”直接显示给用户。
- 语音生成：使用 `FunAudioLLM/CosyVoice2-0.5B` 进行 TTS。
- MP3 输出：将 MP3 采样率固定为服务允许的 `32000Hz` 或 `44100Hz`。
- 发送策略：实现降级链路：

```text
微信气泡语音 SILK
        ↓ 失败
MP3 文件
        ↓ 失败
WAV 文件
        ↓ 失败
文字回复
```

- 统一由 `ReplyOrchestrator` 负责选择回复形式，避免每个功能模块重复实现语音发送逻辑。
- 语音模式开启/关闭由 `VoiceModeHandler` 负责，模式按用户 ID 保存。

### 4. 涉及的技术原理

- **SILK 编码：** SILK 是适合语音通话的有损音频编码格式，微信语音消息除了音频字节，还依赖播放时长、采样率、编码类型等消息元数据。
- **PCM/WAV：** PCM 是未压缩音频采样数据；WAV 通常是在 PCM 前加 RIFF/WAVE 文件头，便于 STT 服务识别采样率、声道数和位深。
- **STT：** Speech-to-Text 将音频特征转换为文本，模型是否能正常工作取决于音频格式、采样率、请求 MIME 类型和 multipart 参数。
- **TTS：** Text-to-Speech 将文本合成为音频，不同格式对采样率有硬性约束，不能只修改文件后缀。
- **降级策略：** 将高风险传输能力和低风险文字能力串联，保证外部服务失败时仍能给用户可见反馈。

### 5. 后续改进

- 启动时检查 SILK 编解码器、TTS、STT 配置并输出可读的健康状态。
- 对音频做格式校验：文件头、字节长度、采样率、声道、播放时长必须一致。
- 记录“生成成功”和“微信客户端可渲染”两个不同状态，避免仅凭 SDK 无异常就打印发送成功。
- 对 STT/TTS 请求增加超时、有限重试和错误分类。
- 将音频生成改为异步任务，避免长文本语音阻塞微信消息轮询线程。

## 二、天气城市、区域和上下文识别不准确

### 1. 遇到的问题、现象和错误关键词

天气功能出现过以下错误：

- 用户询问杭州，解析出的城市却是 `让你是杭州`。
- 用户询问无锡，解析出的城市是 `询一下无锡`。
- 用户先问杭州，再说“那南京呢”，解析成 `杭州那南京`。
- 用户补充“梁溪区”后，查询结果仍显示无锡市，未精确到梁溪区。
- 和风天气返回 gzip 压缩内容时，JSON 解析失败。

典型错误关键词：

```text
city=让你是杭州
city=询一下无锡
text=杭州那南京呢天气
city=杭州那南京
Illegal character ((CTRL-CHAR, code 31))
天气API返回非200
```

`CTRL-CHAR, code 31` 对应 gzip 数据的压缩控制字节，说明程序把压缩二进制内容直接交给了 JSON 解析器。

### 2. 分析过程与思考方式

先把问题分成三层，而不是只修改一个正则表达式：

1. **文本解析层：** 是否从自然语言中正确提取城市、区县和日期。
2. **上下文层：** “那南京呢”“梁溪区”是否应该继承上一轮天气任务。
3. **API 传输层：** 返回体是否被压缩，查询参数是否使用了正确的 location ID。

日志表明，错误城市名是在调用天气 API 之前产生的，所以第一阶段问题是提取器把“查询一下”“让我”“那”等功能词混进了城市名。对“那南京呢”直接拼接上一轮城市又会产生 `杭州那南京`，说明上下文补全没有先识别“替换城市”，而是做了字符串拼接。

区域问题则需要把“城市”和“区域”作为两个逻辑字段处理，不能只把 `无锡梁溪区` 当作一个普通字符串交给地理搜索。正确流程应该是：先识别城市 `无锡`，再识别区域 `梁溪区`，然后使用和风地理查询得到的精确 location ID。

### 3. 解决方案

- 新增 `WeatherHandler`，统一负责：
  - 天气关键词判断。
  - 城市、区县和日期解析。
  - “今天/明天/后天/星期几/具体日期”转换成 `dayOffset`。
  - 和风天气查询结果记录到上下文。
- `ConversationContextService` 负责短句补全，例如上一轮询问天气后，将“无锡呢”识别为新的天气查询，而不是普通聊天。
- 引导任务中，城市缺失或区域信息不完整时先追问用户，而不是立即调用天气 API。
- `WeatherService` 增加 gzip 响应识别和解压，再交给 Jackson 解析。
- 地区查询使用和风返回的 location ID，不直接拼接城市字符串作为最终定位依据。

### 4. 涉及的技术原理

- **意图识别与实体抽取：** “天气查询”是意图，“无锡”“梁溪区”“明天”是实体和参数，二者需要分开处理。
- **上下文解析：** 后续短消息必须结合上一轮意图、城市和日期，而不是单独进行字符串匹配。
- **地理编码：** 城市名称不一定唯一，天气 API 通常先通过地理查询换取 location ID，再使用 ID 查询天气。
- **HTTP 内容编码：** `Content-Encoding: gzip` 表示响应体经过压缩；必须先解压成 UTF-8 JSON，再进行反序列化。
- **防御式解析：** 解析失败时应保留原始请求、解析出的字段和 API 状态码，便于区分“识别错误”和“服务错误”。

### 5. 后续改进

- 将天气请求建模为结构化对象：`city`、`district`、`dayOffset`、`locationId`，避免用拼接字符串传递状态。
- 引入行政区缓存，减少同一城市/区域的重复地理查询。
- 对多个候选地点进行消歧，必要时主动询问用户选择哪个区域。
- 增加天气解析测试样例：

```text
杭州今天天气
可以帮我查一下无锡今天的天气吗
那南京呢
无锡市梁溪区明天天气
我在杭州，查杭州明天的天气
```

- 日志中同时记录 `rawText`、`parsedCity`、`parsedDistrict` 和 `locationId`，但不记录 API Key。

## 三、功能误路由与项目结构耦合

### 1. 遇到的问题、现象和错误关键词

机器人早期把不同类型需求混在一个主流程中，出现过：

- 用户说“没有了谢谢”后，系统仍继续生成海报。
- 用户发送普通聊天或语音请求时，LLM 输出的 `IMAGE_GEN` 标记误触发图片生成。
- 一条消息包含天气、旅游规划和路线图多个需求时，只完成部分需求。
- 业务代码直接调用微信 SDK，语音、文字、图片和文件的发送逻辑重复。

典型错误关键词：

```text
检测到 LLM 请求图片生成
IMAGE_GEN
信息已收齐，正在生成海报
检测到复合需求
功能路由判断
```

其中，`IMAGE_GEN` 不是用户应该看到的内容，而是内部控制标记；“没有了谢谢”被继续送入海报生成流程，说明待完成任务状态、关闭意图和新功能路由的优先级没有正确处理。

### 2. 分析过程与思考方式

先确定消息处理应该是一个明确的状态机，而不是“看到关键词就执行”：

1. 先判断当前用户是否存在待完成任务。
2. 判断新消息是继续该任务、关闭该任务，还是一个全新的需求。
3. 对新需求先做意图分类，再选择对应的 feature handler。
4. 只有在明确判定为图片生成时，才允许执行图片模型。
5. 功能执行完成后，再把结果交给统一回复层，而不是每个模块直接发消息。

这也解释了为什么单纯增加更多关键词不能根治问题：关键词只能解决局部匹配，不能解决任务状态、功能优先级和模型选择之间的关系。

### 3. 解决方案

完成了模块化单体结构迁移：

- `WechatBot`：只保留登录生命周期、消息分发、文本拆分和顶层路由。
- `WechatMessageSender`：隔离微信 SDK 的发送细节。
- `ReplyOrchestrator`：统一文字、MP3、WAV、文字兜底，并清理 `IMAGE_GEN` 等内部标记。
- `ImageHandler`：负责比例解析、图片生成、发送和海报满意度状态。
- `WeatherHandler`：负责天气参数解析、查询和天气上下文。
- `RouteMapHandler`：负责天气参考、结构化路线规划、绘图和发送。
- `FileHandler`：负责文件下载、解析、分析、生成和发送。
- `IncomingMediaHandler`：负责图片、语音、视频输入转换。
- `VoiceModeHandler`：负责按用户切换语音回复模式。
- `GuidedTaskHandler`：统一处理引导任务完成后的天气、图片和路线图执行。

同时保留复合需求拆分机制，使“然后、另外、并且、再帮我”等连接词可以把一条消息拆为多个子请求，逐一经过意图判断和功能处理。

### 4. 涉及的技术原理

- **分层架构：** 入口层不直接依赖外部 SDK 细节，业务功能通过应用服务和适配器完成。
- **领域模块化：** 每个 feature 对应一个完整用户能力，内部封装解析、执行、状态和发送协作。
- **状态机：** 待追问、已收齐、生成中、等待满意度确认、已取消等状态决定下一步动作。
- **路由优先级：** 关闭命令和新功能意图应优先于旧任务继续；模型生成标记只能在明确意图允许时执行。
- **控制标记与用户文本分离：** `IMAGE_GEN` 只作为内部协议，发送前必须清洗，避免内部控制信息泄漏到微信聊天框。
- **适配器模式：** 微信 SDK 被包装在 `WechatMessageSender` 中，便于更换发送方式和编写测试替身。

### 5. 后续改进

- 用结构化 `IntentResult` 替代部分正则判断，统一返回意图、置信度和参数。
- 将“待完成任务”抽象为带类型和状态的任务对象，避免由多个服务分别保存 pending 状态。
- 增加路由测试矩阵，覆盖普通聊天、天气、图片、路线图、文件、语音模式和取消任务。
- 将图片生成、路线图生成、TTS 和文件生成改为异步任务，增加任务 ID、超时和进度状态。
- 增加消息去重和幂等处理，避免微信重复投递导致重复生成图片或文件。
- 将根包中的 `LocalLLMService`、`WeatherService`、`MediaHelper` 进一步归入 `infrastructure` 或对应 feature 包。
